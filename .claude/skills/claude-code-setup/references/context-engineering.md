# Context Engineering 심화

## 핵심 원칙

Context Engineering은 "Claude가 읽게 될 전체 시스템 상태를 어떻게 배치하느냐"를 다룬다.

세 가지 엔지니어링 구분:
- **Prompt Engineering**: 입력창에 뭐라고 쓸지 (현재 요청의 문장)
- **Context Engineering**: Claude가 바깥에서 무엇을 같이 읽을지 (자료와 규칙의 묶음)
- **Harness Engineering**: 어떤 권한과 검증 안에서 움직이게 할지 (실행 환경과 안전장치)

## 항상 상주시키기 좋은 것

- CLAUDE.md (짧게)
- 짧은 공통 규약
- 꼭 필요한 commands
- 프로젝트 핵심 구조

## 가끔만 필요한 것 → 그때 불러오기

- 세부 라이브러리 레퍼런스
- 긴 사례집, 예제 모음
- 오래된 설계 문서

## 길고 무거운 것 → skill이나 supporting file로 분리

- 회사 브랜딩 규칙 전체
- 특정 직무용 상세 템플릿
- 복잡한 운영 절차
- 사례 기반 리뷰 체크리스트

## 토큰을 가장 많이 먹는 5가지

1. 너무 긴 CLAUDE.md와 상시 규칙 파일
2. 범위가 모호한 프롬프트 ("인증 모듈을 개선해 줘")
3. 긴 세션 누적 (이전 대화가 계속 다시 실림)
4. 과도한 도구 출력 (테스트 전체 로그, 검색 결과 수백 줄)
5. 한 세션이 너무 많은 역할을 동시에 맡는 경우

## Tool Output 관리

- 요약된 출력만 보게 한다
- hook이나 wrapper로 tail과 head를 건다
- pass/fail만 남기고 긴 출력을 버린다
- 실패 시 핵심 라인만 남긴다

## Prompt 길이와 품질

세팅이 좋아질수록 프롬프트는 짧아진다. 좋은 프롬프트는 이미 설계된 문맥을 정확히 호출하는 신호다.

나쁜 프롬프트:
```
이번에는 우리 브랜드 톤을 지키고, 지난번 보고서 형식으로, 경쟁사 5개를 조사하고...
```

좋은 프롬프트:
```
`context/`와 `templates/weekly-brief.md`를 먼저 읽고,
지난 7일 기준 경쟁사 브리핑을 만들어.
불명확한 부분은 AskUserQuestion부터 사용해.
```

## Handoff 문서가 세션 압축보다 믿을 만하다

긴 세션은 반드시 압축된다. 자동 압축이 중요한 결정을 잃어버릴 수 있다. 그래서 HANDOFF.md, plan.md, work-log.md 같은 외부 상태 파일을 남긴다.

plan.md: 앞으로 어디로 갈지 적는 지도 (목표, 완료 기준, 순서)
handoff.md: 지금 어디까지 왔는지 남기는 현재 위치 메모 (한 일, 막힌 점, 다음 파일)
